# Internationalization (i18n) & Escaping Standards

Ensure all user-facing strings are translatable and all dynamic output is properly escaped.

## Translation Standards

- **Always use a text domain:** Detect the project's text domain from the plugin header or theme style.css. If unknown, ask the user.
- **Use the correct translation functions:**
  - `__()` — Fetch translated string.
  - `_e()` — Echo translated string.
  - `_n()` — Plural strings.
  - `_x()` — Translated string with context.
- **Escape while translating:**
  - `esc_html__()` — Safe HTML output.
  - `esc_attr__()` — Safe for HTML attributes.
  - `esc_html_e()` — Safe HTML output (echo).
  - `esc_attr_e()` — Safe for HTML attributes (echo).
  - `esc_html_x()` — Safe HTML with context.

## Placeholders & Variables

- **NEVER** concatenate variables inside translation functions.
- **Always** use `printf()` or `sprintf()` with placeholders (`%s`, `%d`).
- **Good:** `printf( esc_html__( 'Hello %s', 'text-domain' ), esc_html( $name ) );`
- **Bad:** `echo __( 'Hello ' . $name, 'text-domain' );`

## Escaping Rules

- **Escape Late:** Escape as close to the point of output as possible.
- **Escape Everything:**
  - `esc_html()` — For plain text.
  - `esc_attr()` — For HTML attributes (id, class, value).
  - `esc_url()` — For URLs (href, src).
  - `esc_js()` — For inline JavaScript strings.
  - `wp_kses()` — For blocks of HTML (use with a permitted tags array).
  - `wp_json_encode()` — For passing data to JS via `wp_localize_script`.

## Prohibited Patterns

- No hardcoded strings in HTML (except for non-user-facing system labels if absolutely necessary).
- No `echo $variable;` without an `esc_*` function prefixing it.
- No `_()` (the PHP native function) — always use the WordPress derivatives.
- No missing text domains in WordPress projects.
